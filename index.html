    • BrixJS aka "bricks"

    Lightweight, simple, but pretty powerful addition to Marionette.js framework

    Project maintained by Denis Davydkov Hosted on GitHub Pages — Theme by mattgraham

    Why?

    No doubts that with Marionette.js you can build great large-scale frontend applications.
    But when you are a large team, you have to implement a lot of screens and your application navigation changes frequently while you want keep everything bookmarkable - that routers and controllers are not so helpful.

    BrixJS (like "bricks") gives you a great opportunity to build your Backbone/Marionette application, but using another vision of navigation and core architecture.

    Idea

    The idea of BrixJS is similar to the one implemented at Google Web Toolkit (GWT).
    "Places" are defined as bookmarkable states, and when a user comes to some place the corresponding "activity" is run. With Marionette layouts and views you define different regions.
    With BrixJs you manage these regions.

    That's it!

    Benefits from using BrixJS:

    • application could be divided to several absolutely independent modules, so their development could be easily paralled
    • view regions are updated independently - some could be re-rendered, some could stay untouched, that leads to better UI-performance
    • infinite hierarchy of view regions and their managers

    Requirements

    This framework is based on

    • Marionette 1.0.0-rc3

    Boilerplate application

    Boilerplate application is using:

    • Underscore 1.4.3 (AMD-version)
    • Backbone 0.9.10 (AMD-version)
    • RequireJS 2.1.2 (could be easily removed)
    • Handlebars 1.0-rc1
    • RequireJS-Handlebars plugin 0.4.0

    Online and compiled version is available here

    To run example application locally:

    npm run-script dev

    or

    node dev-boilerplate.js
